A design system is your colors, spacing, border-radii, and components decided once and used everywhere. One change to a color or spacing value reflects across the entire product.

Most teams use design systems already. But with AI tools writing most of the code now, having a system isn't enough - you need AI to follow it too.

## How to define a design system

### 1. Define your colors once

Pick your brand colors - primary, secondary, error, success, neutrals. Put them in one file like `globals.css` or your Tailwind config. Use names everywhere - `bg-primary`, `text-error` - never a color code.

### 2. Use Tailwind's built-in values - no custom numbers

Paddings, margins, gaps, widths, heights, max-widths - use only what Tailwind gives you.

`p-4`, `m-6`, `gap-4`, `w-full`, `max-w-xl`, `h-12`, `min-h-screen`

No `p-[13px]`, no `gap-[22px]`, no `max-w-[743px]`. Custom numbers break the visual rhythm - sections stop aligning and pages feel stitched together instead of designed.

### 3. Keep components small - UI only, no logic

A Button renders text, handles a click, shows a loading state. No API calls. No data fetching. No business logic inside.

The moment you bake logic into a component, nobody can reuse it - they'll build a new one instead.

### 4. One folder for all components

Button, Input, Card, Badge, Avatar - all in `/components/ui/`. One place to find them, one place to add new ones.

### 5. Build big from small

Need a profile card? Compose it from what you already have.

```tsx
<Card>
  <Avatar src={user.photo} />
  <Text variant="h3">{user.name}</Text>
  <Button variant="secondary">View Profile</Button>
</Card>
```

### 6. Storybook - if your product is big

If the same system is used across your website, blog, and product - set up Storybook so anyone can see every component and how to use it.

## What happens when AI doesn't know your system

You give Cursor a screenshot, connect Figma MCP, or just vibe-code a feature. The code works. But:

- **Random colors** - it used `bg-gray-100` instead of your `bg-primary`
- **Made-up spacing** - it wrote `p-[18px]` trying to match a screenshot pixel-perfect
- **Rebuilt components** - it built a new button from scratch on page itself instead of using existing component from `/components/ui/`

AI writes what looks reasonable _in general_. But general is not your product. Do this across a few features and your system falls apart - random colors, no spacing rhythm, duplicate components, tech debt piling up.

## How to make AI follow your design system

Cursor lets you write a rules file - plain English - that AI reads before writing any code. Like onboarding a developer who reads the instructions every time and never forgets.

Create `.cursor/rules/design-system.mdc` in your project:

```markdown
# Design System

Follow these rules for every file you create or modify.

## Colors
- Only use named colors from our theme (bg-primary, text-error, etc.)
- All colors are defined in globals.css - use those only
- Never use color codes directly (no bg-[#2563EB])

## Spacing
- Only use Tailwind's built-in spacing values
- Padding: p-2, p-3, p-4, p-6, p-8
- Gaps: gap-2, gap-4, gap-6, gap-8
- Widths: max-w-sm, max-w-md, max-w-lg, max-w-xl
- No custom values (no p-[13px], no gap-[22px], no max-w-[743px])

## Components
- Always check src/components/ui/ before building anything new
- If a Button exists, use it - don't build a new one
- If a Card exists, use it - don't build a new one
- Build pages by composing existing components
- Keep components small - no business logic inside
```

With this, AI uses your colors, follows your spacing scale, and reaches for existing components instead of making new ones. Anyone on the team can prompt AI to build features and the output stays consistent - even if they don't know the system by heart.

Five minutes to set up. Keeps every feature - human or AI-built - on system.
